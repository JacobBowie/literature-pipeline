# Independent Codebase Audit — literature-pipeline

**Date:** 2026-07-09
**Scope:** Full repository at `bbe14a8` — 32 top-level scripts (~9.2k LOC), 15 test files (~1.6k LOC, 165 collected tests), `vendor/mathml_to_latex` (~4.7k LOC), docs/CI/config.
**Method:** Read-only static analysis. Twelve independent audit passes (per-subsystem, test-suite value, cross-file duplication, architecture, vendored code, docs/CI/deps), with every removal or defect claim independently adversarially verified before inclusion (130 findings verified: 70 confirmed as filed, 60 confirmed with corrections, 0 refuted outright). No project code was executed. CI status was confirmed against the GitHub Actions run history.

---

## 1. Executive summary

**The verdict on your core suspicion is mixed — and mostly reassuring on tests, less so on structure.**

1. **The 165 tests are ~80% real.** 75 of 94 test functions would genuinely fail if the behavior they name regressed. Only ~24 of 165 collected tests are tautological, trivial, or redundant. The real problem is subtler and worse: **six regression tests test private mirrors of production logic**, so the exact bugs they were written to pin (sweep rename collision, migrate DOI dedup, DuckDB upsert abstract-preservation) could regress today with the suite fully green. The single most valuable invariant in the repo — *re-indexing must not clobber ~9,800 CrossRef-enriched abstracts* — has **zero** real coverage.

2. **The dominant bloat is not dead code — it is copy-paste infrastructure.** The shared-module story is inverted: `lit_util.py` (143 lines) holds only audit-remediation primitives while the actual cross-cutting utilities are pasted per script — the UTF-8 stdout shim ×26 files, the `LITPIPE_EMAIL` default ×15, `projects.json` resolution ×9–11 (in **two divergent conventions**), the fitz DOI-from-PDF reader ×6, the streaming PDF downloader ×3, the DuckDB open-with-retry ×3, NCBI idconv ×3. Roughly **630 net lines** are deletable by consolidation alone, and the duplication is actively harmful: several copies have already drifted into live bugs (§2).

3. **CI on `main` is red and has been since June 6.** The last two runs (`467e153` Jun 6, `bbe14a8` Jul 8) concluded `failure`; the last green run was May 13. Cause (verified statically): `make_convergence_hero.py` — a personal-website figure generator with hardcoded `~/Projects/jacobbowie-site` paths, unguarded `pandas`/`matplotlib` imports declared in no requirements file, and no `--help` — cannot survive the CI smoke step. It should be deleted from this repo (−146 lines, fixes CI).

4. **The vendored MathML tree is NOT bloat — keep it.** All eight lookup tables in `syntax.py` (3,308 lines) are live on the conversion path from the single entry point (`jats_to_text.py:42`); total dead vendor code is ~22 lines. PyPI's `mathml-to-latex` is the same code frozen at v1.0.0 and would drop your two local patches. One compliance gap: the vendored copy ships **no upstream MIT license text**, which the MIT license requires.

5. **Realistic optimization envelope:** ~750 net lines removable at low risk, ~1,300–1,600 with the moderate/judgment tiers (≈15% of non-vendor LOC), while *fixing* roughly a dozen real defects — because in this codebase the duplication is where the bugs live. (The raw sum of per-finding estimates was ~2,200; that number double-counts overlapping findings and is corrected here.)

## 6. Structural reorganization

The repo is a de-facto library masquerading as 32 flat scripts: 16 carry `sys.path.insert` hacks so siblings can import each other; `lit_util` is imported by 17, `ris_emit` by 6; orchestration is subprocess-glued via `sys.executable`. Two options:

**Option A — Conservative (recommended first, ~230 lines):** keep the flat layout; execute the §4 consolidation into `lit_util`. Zero migration risk, most of the value, no README/CI churn. Every later option remains open.

**Option B — Package (~350 lines + kills the sys.path hacks):** `litpipe/` with `config.py` (stdout shim, EMAIL/UA, `PROJECTS_ROOT`/`DB_PATH`/`CONFIG_PATH` once), subpackages `fetch/`, `cite/`, `extract/`, `queue/`, and a `python -m litpipe <cmd>` argparse dispatcher; `pyproject.toml` with a console script. Migration is mechanical (~15 subprocess call sites, test imports, CI loop) but touches everything at once. Do it as a single dedicated change *after* Option A, or not at all — A captures most of the LOC value.

The config story should end up: one module defining paths/identity, `projects.json` gaining optional `projects_root`/`db_path` keys (env-overridable). Today the portfolio root is resolved three different ways across 15 files (`~/Projects` ×7, POSIX-broken `r"~\Projects"` ×7, `HERE.parent.parent` in sweep) and the DB path is hardcoded 6×.

## 8. Coverage disclosure

An audit is only as good as what it admits it didn't do. Not deep-reviewed: `backfill_fulltext.py`'s main/report logic (~150 LOC), `extract_tables.py` (144), `fetch_figures.py`'s report path, `seed_queue_from_top_candidates.py` (130), and `jats_to_text.py`'s core `parse_jats` (~250 LOC — the parser feeding the whole extraction layer; only its smoke-test and counters were audited). Not systematically audited: type-hint hygiene, bare-`except` consistency, print-vs-stderr conventions. Security spot-checks came back clean (DuckDB queries parameterized; all ~20 `requests` calls carry explicit timeouts), but filename-sanitization strength against remote-metadata-derived names was not exhaustively traced. Fourteen low-ranked findings passed finder review but were not independently verified (marked in the appendix). CI's live failure logs were not inspected (only run conclusions); the statically-derived cause (B1) is high-confidence but the June 6 failure could additionally involve other steps.
